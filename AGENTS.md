# LubanVision 项目协作规则

本文件是新任务进入仓库后的默认工作约定。目标是让每个任务都形成“读取现状、验证设备、
推进唯一任务、留存证据、同步文档、明确下一步”的闭环，而不是只修改代码。

## 每次新任务开始

1. 先读取`README.md`、`docs/03_progress_status.md`、`docs/04_issue_log.md`、
   `docs/05_test_plan.md`和`docs/08_execution_playbook.md`，再查看与当前任务直接相关的设计或
   安装记录。
2. 执行`git status -sb`，确认分支、已有修改和远端差异；用户已有变更不得覆盖。
3. 按`docs/01_quick_start.md`核对Mac、WSL和鲁班猫的网络、SSH及当前任务涉及的设备。
   只读检查优先；机械动作、PWM输出和设备解绑必须有明确任务目的与安全边界。
4. 从进度文档中的“当前唯一实施任务”恢复，不重复已经通过的里程碑，不顺手扩展下一阶段。

## 推进与验收

1. 先复现或建立基线，再做最小范围修改。
2. 代码、配置、设备状态和文档结论必须由实际命令、日志或现场观察支持。
3. 修改后执行与风险匹配的构建、自动测试和真实设备检查；失败时保留原始错误，不把“基本
   可用”写成“已完成”。
4. 正式验证证据保存在`artifacts/YYYYMMDD/Mxx-short-name/`，至少包括`commands.txt`、
   `environment.txt`、`result.md`和必要的`logs/`。
5. 测试结束必须确认临时进程、PWM、设备绑定和其他可变状态已清理或恢复，并再次检查设备
   在线状态。

## 每次任务结束前的文档闭环

不要只更新一份进度文档。逐项判断并同步所有受到本次事实影响的文件：

- `README.md`：仓库能力、已完成阶段和下一任务的高层摘要。
- `docs/01_quick_start.md`：新线程的恢复入口、可靠命令、设备地址和故障恢复方法。
- `docs/02_design.md`：架构、接口或安全策略发生变化时更新；不记录流水账。
- `docs/03_progress_status.md`：里程碑状态、当前事实、证据位置和下一张唯一任务。
- `docs/04_issue_log.md`：问题的现象、根因、修复、回归和预防措施。
- `docs/05_test_plan.md`：实际测试日期、状态、指标和对应证据。
- 平台安装记录：环境、软件源、包版本或设备映射变化时更新。
- `docs/08_execution_playbook.md`：可复用工作方法以及“当前恢复点”。

更新后使用全文搜索检查旧状态、旧日期、旧下一任务和互相矛盾的结论。只有当实现、验证证据、
测试计划、问题记录、进度和恢复入口彼此一致时，任务才算真正收尾。

## Git收尾

提交前检查`git diff --check`、`git diff --stat`和实际差异，只暂存本任务文件。完成验收且工作树
范围清楚时，将代码、证据和对应文档放在同一个本地提交中。除非用户明确要求，不自动推送、
创建PR或覆盖远端历史。最终汇报设备状态、测试结果、提交状态、证据位置和下一任务。

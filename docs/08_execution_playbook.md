# LubanVision半小时执行手册

## 1. 目的

本手册把项目拆成模型一次约30分钟可完成的任务卡，减少重复探索、遗漏和无效上下文。
它既用于当前项目复盘，也作为后续Linux、ROS 2和机器人项目的执行模板。

30分钟是**主动分析与实施预算**，不是对网络下载、硬件采购或稳定性测试墙钟时间的承诺。
遇到外部等待时，任务卡应在30分钟内完成环境确认、启动可恢复操作、记录断点和下一步。

## 2. 单任务原则

一次执行只处理一张`Mxx`任务卡，并遵守：

1. 开始前读取进度、当前问题、相关安装记录和`git status`。
2. 用一句话写清本次唯一目标，不顺手扩展下一阶段功能。
3. 先复现和留证，再修改环境或代码。
4. 修改后必须执行与风险匹配的最小测试。
5. 测试未通过时保持里程碑“进行中”，不能用“基本完成”代替验收。
6. 结束前先更新文档，再决定是否提交代码。
7. 密码、令牌、Wi-Fi密钥和私人证书不得进入文档、日志或Git。

## 3. 任务卡模板

每张卡在执行前应具备以下字段：

```text
任务ID：Mxx
唯一目标：
预计主动工作：30分钟以内
前置条件：
输入：
允许修改：
禁止操作：
执行步骤：
验收命令：
完成条件：
失败条件：
日志位置：
关联测试：
关联问题：
文档更新：
下一任务：
```

## 4. 30分钟节奏

| 时间 | 工作 | 必须产出 |
|---|---|---|
| 0-5分钟 | 读取状态、确认设备和复现 | 当前事实与唯一目标 |
| 5-10分钟 | 选择最小方案和回滚边界 | 执行步骤与风险 |
| 10-22分钟 | 实施代码或配置 | 可检查的变更 |
| 22-27分钟 | 构建、测试或采样 | 原始结果和日志 |
| 27-30分钟 | 更新文档和Git状态 | 结论、问题、下一任务 |

如果第10分钟仍无法定义最小方案，应停止实施，改为调查任务卡并输出决策记录。如果第22分钟
仍在不可控下载或测试中，应记录PID、日志和恢复命令，不得盲目开启第二个相同进程。

## 5. 状态定义

- `未开始`：前置条件可能已具备，但尚未执行。
- `进行中`：已产生变更或证据，但完成条件未全部满足。
- `已完成`：验收命令通过，产物和文档存在。
- `受阻`：缺少硬件、认证或外部服务；必须有问题编号和解除条件。
- `跳过`：经记录的设计决策确认不属于当前首版。

不得把“命令运行过”“包安装过”“代码写完了”单独视为已完成。完成必须同时满足行为验证、
测试结果和文档记录。

## 6. 日志与产物

正式验证使用以下结构，避免只留在易失的`/tmp`：

```text
artifacts/
  YYYYMMDD/
    Mxx-short-name/
      commands.txt
      environment.txt
      result.md
      logs/
```

- `commands.txt`记录实际执行命令，删除密码和令牌。
- `environment.txt`记录主机、系统、架构、ROS版本和代码提交。
- `result.md`记录预期、实际、结论、指标和关联问题。
- 大型rosbag、视频和构建产物不直接提交Git；文档记录其存放位置和校验值。
- `/tmp`只用于冒烟测试；正式验收前必须复制到持久目录。

## 7. 文档职责

| 文档 | 更新时机 | 内容 |
|---|---|---|
| `03_progress_status.md` | 每张任务卡结束 | 状态、事实、当前唯一任务 |
| `04_issue_log.md` | 发现或关闭问题 | 复现、根因、修复、回归和预防 |
| `05_test_plan.md` | 每次测试后 | 测试状态、指标和日志位置 |
| 安装记录 | 环境变化后 | 软件源、包版本、命令和平台差异 |
| `02_design.md` | 接口或架构变化后 | 决策及其影响，不记录流水账 |
| Git提交 | 一组通过验收的变更后 | 代码与对应文档一起提交 |

## 8. Git提交与推送

本项目固定远端为：

```text
https://github.com/lc285800/rk3588-camera-robot.git
```

每张任务卡完成后按以下顺序收尾，禁止凭仓库目录名猜测远端，也不要在未检查差异时直接
`git add -A`：

```bash
# 1. 确认当前分支、工作树和远端
git status -sb
git remote -v
git branch -vv

# 2. 检查本次范围、格式和敏感信息
git diff --check
git diff --stat
git diff
rg -n -i 'password|token|secret|BEGIN .* PRIVATE KEY' <本次文件> || true

# 3. 只暂存当前任务相关文件并复核
git add <明确的文件或目录>
git diff --cached --check
git diff --cached --stat

# 4. 提交并推送
git commit -m "Complete Mxx short description"
git push

# 5. 确认本地与远端一致
git status -sb
git log -1 --oneline --decorate
```

如果新工作区没有`origin`，先用GitHub CLI确认仓库身份，再只配置一次：

```bash
gh auth status
gh repo view lc285800/rk3588-camera-robot \
  --json nameWithOwner,url,defaultBranchRef,isPrivate
git remote add origin https://github.com/lc285800/rk3588-camera-robot.git
git push -u origin main
```

2026-07-16首次发布时，GitHub目标仓库为空，本地已有M00-M13完整提交历史，因此将本地
`main`直接作为远端`main`首次推送，并建立`main -> origin/main`跟踪关系。后续普通提交只需
`git push`。如果远端已存在不一致历史、工作树包含无关修改，或目标不是上述仓库，应停止并
先确认，不得使用强制推送覆盖远端。

## 9. 问题处理规则

出现错误时按固定顺序处理：

1. 保存完整错误和返回码。
2. 记录环境与最小复现，确认是否由旧缓存或并发进程造成。
3. 查看项目已有问题和安装记录，避免重复调查。
4. 只做一个最小修复，然后执行原复现命令。
5. 记录无效尝试及原因，防止下次再次消耗时间。
6. 涉及密钥、软件源、硬件电压和机械动作时，先验证安全边界。

## 10. 本项目已沉淀的经验

- LubanCat-5与LubanCat-5-V2镜像和设备树不能混用。
- ROS旧固定密钥会过期，应优先使用官方`ros2-apt-source`配置包。
- 代理应按域名配置；国内镜像直连，外部ROS源走代理。
- 代理改善可用性不等于提高吞吐，必须用实际下载数据判断。
- Rockchip多媒体PPA可能与Ubuntu FFmpeg开发包精确版本冲突，不能贸然降级运行库。
- USB摄像头节点不能只凭`/dev/video0`假设，必须先读取设备名称和格式。
- `colcon test`显示0项测试不是通过，需要修复测试发现配置。
- 测试报告可能残留旧结果，回归前核对源码、XML时间戳和构建缓存。
- `pkill -f`可能匹配包含目标字符串的父级远端命令；先独立查询PID，再按精确PID停止进程。
- ROS原始图像应使用传感器数据QoS；可靠QoS可能在大消息和慢订阅者下形成背压。
- Git操作前先核对`origin`和GitHub仓库身份；没有远端不代表可以凭名称直接新建或推送。

## 11. 当前恢复点

M00-M14已经完成，阶段P4软件闭环通过。M15 SoC PWM与供电盘点仍保留最终电源复测；物理Pin 32
的`GPIO4_B6 / PWM13_M1`、50Hz周期、1.5ms中位脉宽和安全关闭的软件测试已通过。2026-07-20
真实SG90在1.45-1.55ms范围完成短时硬件冒烟，用户确认轻微转动且RK未重启。当前12V 1.5A
输入低于官方规格，SG90又暂用板载共享5V，因此M15仍进行中。恢复时先升级为12V 3A输入，
或改用独立稳压5V 2A以上舵机电源并共地；供电复测通过前不得扩大动作范围。
`lubanvision_hardware`受限sysfs PWM核心已完成双端测试，平台YAML的1.45-1.55ms范围只代表
已有冒烟值。恢复硬件测试时使用该核心的限位和清理语义，但正式供电复测前仍不得启用扫描
或扩大脉宽。
2026-07-20已按用户指示启动M16受限三点复验，正式安全核心执行成功且系统健康；用户确认
1.45ms向左、1.50ms居中、1.55ms向右。临时85/90/95度名义映射已通过WSL/RK各38项硬件包
测试和各82项工作区回归。当前唯一后续硬件任务仍是升级供电并复测M15，再实际量角和验证
M16机械边界；不得因本轮成功扩大脉宽。
